# 设计决策记录 (ADR)

## 概述

本文档记录 sfsEdgeStore 项目的重要设计决策，包括技术选型、架构设计等关键决策。

## ADR-001: 选择 LevelDB 作为底层存储

**状态**: 已采纳  
**日期**: 2025-01-15  
**决策者**: 架构团队

### 上下文

需要选择一个高性能的嵌入式键值存储数据库，用于边缘环境下的时间序列数据存储。

### 备选方案

1. **LevelDB**
   - 优点: 高性能、嵌入式、Go 语言支持好
   - 缺点: 单线程压缩、不支持分布式

2. **RocksDB**
   - 优点: 多线程、更好的压缩、更高性能
   - 缺点: CGO 依赖、部署复杂

3. **BadgerDB**
   - 优点: 纯 Go 实现、LSM-tree、事务支持
   - 缺点: 相对较新、社区较小

4. **BoltDB**
   - 优点: 纯 Go、ACID 事务、简单
   - 缺点: 写入性能较低、不适合高写入场景

### 决策

选择 LevelDB 作为底层存储引擎。

### 理由

1. **性能平衡**: LevelDB 在读写性能上达到良好平衡
2. **成熟稳定**: Google 开源，经过大量生产验证
3. **Go 支持**: 有成熟的 Go 绑定 (goleveldb)
4. **部署简单**: 纯 Go 实现，无 CGO 依赖
5. **边缘友好**: 资源占用低，适合边缘设备

### 后果

- 需要自己实现一些高级功能（如批量操作优化）
- 单线程压缩在大数据量下可能成为瓶颈
- 未来可以考虑封装存储层，方便切换到其他引擎

## ADR-002: 使用 MQTT 作为消息协议

**状态**: 已采纳  
**日期**: 2025-01-20  
**决策者**: 架构团队

### 上下文

需要选择一个轻量级的消息协议，用于与 EdgeX Foundry 和其他物联网设备通信。

### 备选方案

1. **MQTT**
   - 优点: 轻量级、广泛支持、低带宽、QoS 支持
   - 缺点: 不保证消息顺序（依赖 broker）

2. **AMQP**
   - 优点: 功能丰富、可靠、灵活
   - 缺点: 较重、资源消耗大

3. **HTTP/REST**
   - 优点: 简单、通用、易于调试
   - 缺点: 开销大、不适合推送

4. **WebSocket**
   - 优点: 双向通信、低延迟
   - 缺点: 连接管理复杂、物联网设备支持有限

### 决策

选择 MQTT 3.1.1 作为主要消息协议。

### 理由

1. **物联网标准**: 物联网领域事实标准
2. **EdgeX 兼容**: EdgeX Foundry 原生支持
3. **轻量高效**: 协议头小，适合低带宽环境
4. **QoS 支持**: 提供不同级别的消息传递保证
5. **广泛支持**: 几乎所有物联网平台和设备都支持

### 后果

- 需要部署 MQTT broker（如 Mosquitto）
- 需要处理消息重发和去重
- 可以考虑在未来添加 MQTT 5.0 支持

## ADR-003: 分层架构设计

**状态**: 已采纳  
**日期**: 2025-01-25  
**决策者**: 架构团队

### 上下文

需要设计一个清晰的系统架构，支持边缘计算场景下的可维护性和可扩展性。

### 备选方案

1. **分层架构**
   - 优点: 清晰、易维护、职责分离
   - 缺点: 可能增加层级开销

2. **微服务架构**
   - 优点: 独立部署、扩展性好
   - 缺点: 复杂度高、资源消耗大（不适合边缘）

3. **单体架构**
   - 优点: 简单、部署容易
   - 缺点: 耦合度高、扩展性差

4. **事件驱动架构**
   - 优点: 松耦合、可扩展性好
   - 缺点: 调试困难、事务处理复杂

### 决策

采用分层架构，分为：
- 接入层 (MQTT/HTTP)
- 业务层 (队列、处理、聚合)
- 数据层 (存储、索引)
- 配置层

### 理由

1. **清晰分离**: 每层职责明确
2. **边缘友好**: 单体部署，资源占用低
3. **可维护性**: 便于理解和修改
4. **可测试性**: 每层可以独立测试
5. **演进路径**: 未来可以拆分出微服务

### 后果

- 需要严格遵守分层原则
- 层间通信需要明确接口
- 避免跨层调用

## ADR-004: 设备名称格式化策略

**状态**: 已采纳  
**日期**: 2025-02-01  
**决策者**: 架构团队

### 上下文

EdgeX Foundry 设备名称长度不固定，需要统一格式以便存储和索引。

### 备选方案

1. **固定长度格式化 (64 字符)**
   - 优点: 索引效率高、存储一致
   - 缺点: 可能丢失信息

2. **哈希映射**
   - 优点: 固定长度、无信息丢失
   - 缺点: 需要维护映射表、无法直接查询

3. **保留原样**
   - 优点: 简单、信息完整
   - 缺点: 索引效率低、存储不一致

4. **截断 + 哈希后缀**
   - 优点: 保留前缀、避免冲突
   - 缺点: 实现稍复杂

### 决策

采用固定长度 64 字符格式化，不足补零，超出截断。

### 理由

1. **索引优化**: 固定长度键查询效率更高
2. **实现简单**: 无需额外映射表
3. **实际够用**: 64 字符对于大多数场景足够
4. **LevelDB 友好**: LevelDB 对固定长度键有优化

### 后果

- 超长设备名可能被截断
- 需要在用户文档中说明
- 未来可以考虑添加设备别名机制

## ADR-005: 批量处理和队列设计

**状态**: 已采纳  
**日期**: 2025-02-05  
**决策者**: 架构团队

### 上下文

高写入场景下，需要优化数据库写入性能，减少 I/O 操作。

### 备选方案

1. **内存队列 + 批量写入**
   - 优点: 减少 I/O、提高吞吐量
   - 缺点: 有数据丢失风险

2. **WAL (Write-Ahead Log)**
   - 优点: 数据安全、可恢复
   - 缺点: 增加写入延迟、磁盘占用

3. **同步写入**
   - 优点: 数据安全、简单
   - 缺点: 性能低、I/O 频繁

4. **混合方案**
   - 优点: 平衡性能和安全
   - 缺点: 实现复杂

### 决策

采用内存队列 + 批量写入 + 可配置的持久化策略。

### 理由

1. **性能优先**: 边缘场景通常追求高吞吐量
2. **可配置**: 用户可以根据需求调整
3. **LevelDB 自带 WAL**: LevelDB 本身有 WAL，提供一定保护
4. **批量优化**: 批量写入显著提高 LevelDB 性能

### 配置选项

```json
{
  "queue": {
    "batchSize": 100,
    "flushInterval": 1000,
    "maxQueueSize": 10000
  }
}
```

### 后果

- 崩溃时可能丢失最多 batchSize 条数据
- 需要监控队列长度
- 可以考虑添加可选的 WAL 增强

## ADR-006: API 设计风格

**状态**: 已采纳  
**日期**: 2025-02-10  
**决策者**: 架构团队

### 上下文

需要设计 RESTful API，提供数据查询和管理功能。

### 备选方案

1. **RESTful JSON API**
   - 优点: 简单、广泛支持、易调试
   - 缺点: 开销较大

2. **GraphQL**
   - 优点: 灵活、按需查询
   - 缺点: 复杂、过度优化

3. **gRPC**
   - 优点: 高性能、类型安全
   - 缺点: 调试困难、浏览器支持有限

4. **混合方案**
   - 优点: 兼顾不同场景
   - 缺点: 维护成本高

### 决策

采用 RESTful JSON API 作为主要接口。

### 理由

1. **简单易用**: 开发者熟悉，易于集成
2. **调试方便**: 可以用 curl、Postman 等工具
3. **广泛支持**: 所有编程语言都有良好支持
4. **文档友好**: 易于生成和阅读 API 文档

### API 设计原则

- 使用名词复数形式 (`/api/readings`)
- 使用标准 HTTP 方法 (GET, POST, PUT, DELETE)
- 使用正确的 HTTP 状态码
- 提供分页和过滤
- 返回一致的 JSON 格式

### 后果

- 对于超高性能场景，可以考虑添加 gRPC 接口
- 需要完善 API 文档
- 考虑添加 OpenAPI/Swagger 规范
